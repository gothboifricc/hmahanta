Data Analysis of Failed Orders

A structured analysis using DAX calculations and data visualisation techniques.

Introduction

Download Dataset

This data project has been used as a take-home assignment in the recruitment process for the data science positions at Gett.

Gett, previously known as GetTaxi, is an Israeli-developed technology platform solely focused on corporate Ground Transportation Management (GTM). They have an application where clients can order taxis, and drivers can accept their rides (offers). At the moment, when the client clicks the Order button in the application, the matching system searches for the most relevant drivers and offers them the order. In this task, we would like to investigate some matching metrics for orders that did not completed successfully, i.e., the customer didn't end up getting a car.


Tasks Assigned


1. Distribution of Orders by Failure Reason

  • Build up distribution of orders according to reasons for failure:
    • Cancellations before and after driver assignment.
    • Reasons for order rejection.
  • Analyze the resulting plot.
  • Identify which category has the highest number of orders.

2. Failed Orders by Hour

  • Plot the distribution of failed orders by hours.
  • Check for trends showing an abnormally high proportion of failures in certain hours.
  • Determine which hours have the highest failure rates and analyze possible reasons.

3. Average Time to Cancellation

  • Plot the average time to cancellation with and without a driver, categorized by hour.
  • Remove any outliers from the dataset for better accuracy.
  • Analyze the plot to draw meaningful conclusions.

4. Distribution of Average ETA by Hour

  • Plot the distribution of average Estimated Time of Arrival (ETA) by hours.
  • Interpret the plot and explain observed patterns.


Data Description


We have two datasets: data_orders and data_offers, both stored in CSV format.


1. data_orders Dataset

This dataset contains detailed order information with the following columns:

  • order_datetime – Time of the order.
  • origin_longitude – Longitude of the order's origin.
  • origin_latitude – Latitude of the order's origin.
  • m_order_eta – Estimated time before order arrival.
  • order_gk – Unique order number.
  • order_status_key – Status of the order, mapped as follows:
    • 4 – Cancelled by client.
    • 9 – Cancelled by system (rejected).
  • is_driver_assigned_key – Indicates whether a driver has been assigned.
  • cancellation_time_in_seconds – Time (in seconds) before cancellation.

2. data_offers Dataset

A simple mapping dataset with two columns:

  • order_gk – Order number, linked to the data_orders dataset.
  • offer_id – Unique offer ID.

Data Preparation


We worked with two datasets: data_orders and data_offers. The primary dataset, data_orders, contained key columns such as order timestamps, order status, driver assignment status, and cancellation times. Before proceeding with analysis, we handled missing values and transformed necessary columns for meaningful insights.


Handling Missing Data


m_order_eta (Estimated Time of Arrival):

  • Replaced blank values with the average ETA for a more complete dataset.
  • Alternatively, excluded blank values from calculations when necessary.
Filled_m_order_eta = IF(ISBLANK(data_orders[m_order_eta]), AVERAGE(data_orders[m_order_eta]), data_orders[m_order_eta])

cancellation_time_in_seconds:

  • Replaced blanks with 0, assuming an instant cancellation.
  • Excluded blanks when calculating averages to avoid skewing results.
Filled_Cancellation_Time = IF(ISBLANK(data_orders[cancellation_time_in_seconds]), 0, data_orders[cancellation_time_in_seconds])

Task 1: Distribution of Orders by Failure Reason


To classify failure reasons, we created calculated columns using SWITCH(...) to handle multiple conditions.


reason_failure = SWITCH([order_status_key], 4, "Cancelled by Client", "Cancelled/Rejected by System")
before_after_assigned = SWITCH([is_driver_assigned_key], 0, "NOT ASSIGNED", "WAS ASSIGNED")

We visualised this data using a bar chart.




Findings:

  • Most orders were cancelled by clients (~68% of total cancelled orders, i.e., 7307 orders).
  • Out of these 7307 cancelled orders, ~39% had drivers assigned.
  • System cancellations (~32% of total cancelled orders, i.e., 3406 orders) had no drivers assigned.

Task 2: Failed Orders by Hour


We analyzed failure trends by extracting the hour from order_datetime using the HOUR(...) function in DAX.


Order_Hour = HOUR('data_orders'[order_datetime])

We then plotted a line chart of cancelled orders by hour.




Findings:

  • Highest failure rates occur between 7-10 AM and 8-12 PM, mainly due to client cancellations.
  • System cancellations peak at 3 AM and 11 PM, likely due to system maintenance.

Task 3: Average Time to Cancellation


We removed outliers and adjusted cancellation times using the following logic:


transformed_cancellation_time = IF(data_orders[cancellations_time_in_seconds] < 3, 0, data_orders[cancellations_time_in_seconds])

We can use a bar chart to visualise this data, helping us determine patterns in cancellation timing.




Findings:

  • Highest average cancellation times occur at 11 AM - 1 PM, with peaks at 5 AM and 8 PM.
  • Shorter cancellation times during 2-4 AM and 9-11 PM indicate smoother operations.

Task 4: Distribution of ETA by Hour


We analyzed ETA distributions and calculated adjusted values:


transformed_m_order_eta =
VAR AvgETA = CALCULATE(AVERAGE(data_orders[m_order_eta]), ALL(data_orders))
RETURN IF(data_orders[m_order_eta] < 60, AvgETA, data_orders[m_order_eta])

Now we compare average ETA values across different hours using a line chart.




Findings:

  • Highest ETA occurs at 8-9 AM and 4-6 PM (peak traffic hours).
  • Lowest ETA times are 4-6 AM, 10 AM-12 PM, and 7-9 PM.
  • Optimizing scheduling and routing during peak hours can reduce delays.

Conclusion


This approach allowed to uncover key insights into order failures. By using DAX Queries, handling missing data, and visualizing trends in Power BI, a deeper understanding of cancellation patterns and their potential causes was achieved. This methodology can be applied to similar datasets for further business intelligence applications.
